# Contributing Guide

Thank you for your interest in contributing to this project.

## How to Contribute

Contributions are welcome through:

* Bug reports
* Feature requests
* Documentation improvements
* Code contributions
* Test improvements
* Performance optimisations

Before starting significant work, open an issue to discuss the proposed change.

---

## Development Workflow

1. Fork the repository.
2. Create a feature branch.

```bash
git checkout -b feature/short-description
```

3. Make your changes.
4. Run tests and verify the build.
5. Commit your changes.
6. Push your branch.
7. Open a Pull Request.

---

## Commit Messages

Write clear and descriptive commit messages.

Examples:

```text
feat: added <problem-name.cpp>
fix: optimised the code for better efficieny
docs: update build instructions
test: add unit tests for validator
refactor: simplify configuration loader
```

---

## Coding Standards

Please follow the project's coding standards:

* Write readable and maintainable code.
* Prefer simplicity over complexity.
* Avoid unnecessary dependencies.
* Keep functions focused on a single responsibility.
* Remove unused code and imports.

---

## Testing

Before submitting changes:

* Ensure the project builds successfully.
* Run all tests.
* Add tests for new functionality whenever possible.
* Verify that existing functionality is not broken.

---

## Documentation

Documentation updates should accompany relevant code changes.

Examples:

* New feature → update user documentation.
* New configuration option → update configuration guide.
* API changes → update API documentation.

---

## Pull Requests

A pull request should:

* Address a single concern or feature.
* Include a clear description of changes.
* Reference related issues when applicable.
* Pass all automated checks.
* Include documentation updates only if needed.

Large pull requests may be requested to be split into smaller, reviewable changes.

---

## Reporting Issues

When reporting an issue, include:

* Project version
* Operating system
* Steps to reproduce
* Expected behavior
* Actual behavior
* Relevant logs or error messages
